# review-diff スキル

git diff を取得し、品質チェックを行い、OK/NG 判定を日本語で出力する。すべての出力は日本語で行うこと。

## 引数の処理

`$ARGUMENTS` を解析:

- `focus:<area>` → レビュー重点領域（例: `focus:security`, `focus:performance`）（任意）
- `spec:<path>` → 照合する Spec ファイルの絶対パス（任意）
- 引数なし → 汎用レビュー

## 処理フロー

1. Bash で `git diff` を取得（ステージ済み: `git diff --cached`, 未ステージ: `git diff`）
2. diff が空の場合は「レビュー対象の変更がありません。」と通知して終了
3. **Spec の自動検出**（ステップ3a参照）
4. 以下の観点でレビューを実施
5. **結果の保存**（ステップ5a参照）

### ステップ3a: Spec 自動検出

`spec:` が明示的に指定されていない場合、以下の順序で自動検出を試みる:

1. Bash で現在のブランチ名を取得（`git branch --show-current`）
2. ブランチ名にプロジェクト名が含まれる場合（例: `feature/homepage-renewal/hero-image`）、`/Users/apple/YourOS/Projects/<project>/Specs/` を Glob で検索
3. 直近のコミットメッセージ（`git log -1 --format=%s`）にプロジェクト名や Spec slug が含まれる場合も同様に検索
4. 候補が1つの場合 → 自動採用し「Spec を自動検出しました: (パス)」と通知
5. 候補が複数の場合 → 一覧表示してユーザーに選択を求める
6. 候補が0の場合 → Spec なしでレビュー続行

`spec:` が明示的に指定されている場合は Read で Spec ファイルを読み込み、`## 受入条件` を抽出。

### ステップ5a: レビュー結果の保存

レビュー完了後、結果を `/Users/apple/YourOS/Scratch/Reviews/review-YYYY-MM-DD--N.md` に保存する:

- `N` は同日のレビュー連番（1, 2, 3...）。既存ファイルを Glob で確認してインクリメント
- 保存内容にはレビュー日時、対象ブランチ、判定結果、チェック結果一覧を含める
- 保存後「レビュー結果を保存しました: (パス)」と通知

## レビュー観点

### 必須チェック（全レビューで実施）

| チェック項目 | 内容 |
|-------------|------|
| セキュリティ | ハードコードされたシークレット、SQL インジェクション、XSS、コマンドインジェクション |
| エラーハンドリング | 未処理の例外、エラーの握りつぶし |
| 型安全性 | any の乱用、型アサーションの不適切な使用（TypeScript の場合） |
| 命名規則 | 一貫性のない命名、略語の乱用 |
| デッドコード | 使用されていないインポート、到達不能コード |

### focus 指定時の追加チェック

| focus | 追加チェック |
|-------|-------------|
| security | 認証・認可のバイパス、入力バリデーション不足、CORS設定、依存パッケージの既知脆弱性 |
| performance | N+1クエリ、不要な再レンダリング、メモリリーク、大量ループ内のI/O |
| accessibility | aria 属性、キーボード操作、コントラスト、スクリーンリーダー対応 |
| testing | テストカバレッジ、エッジケース、モック適切性、テスト名の明瞭さ |

### プロジェクト固有ルールの参照

プロジェクトが特定できた場合、以下を追加で参照:

1. `/Users/apple/YourOS/.company/engineering/CLAUDE.md` の技術基準
2. プロジェクトの `Context/` フォルダにあるコーディング規約
3. プロジェクトの既存コードのパターン（命名規則、構造の一貫性）

### Spec 照合（Spec が検出/指定された場合）

Spec の受入条件の各項目について:
- 条件を満たす変更が diff に含まれているか → 合格
- 条件に関連する変更が不足している → 警告
- 条件に反する変更がある → 不合格

## 出力フォーマット

```
== コードレビュー結果 ==

判定: OK / NG

### サマリー
変更ファイル: N個
追加行: +N / 削除行: -N
対象ブランチ: branch-name

### チェック結果
- [合格] セキュリティ: ハードコードされたシークレットなし
- [警告] エラーハンドリング: catch ブロックでエラーを無視している箇所あり (file.ts:42)
- [不合格] 型安全性: any が3箇所で使用 (file.ts:10, 25, 78)

### 受入条件の照合（Spec 検出/指定時のみ）
- [合格] 条件1: ログインフォームが表示される
- [警告] 条件2: バリデーションメッセージの実装が不完全
- [不合格] 条件3: パスワードリセットリンクが未実装

### 推奨アクション
1. file.ts:42 のエラーハンドリングを修正
2. any を具体的な型に置換

判定根拠: 不合格が1件以上 → NG / 不合格なし → OK
```

## 判定ロジック

- **OK**: 不合格が0件（警告は許容）
- **NG**: 不合格が1件以上
