# review-diff スキル

git diff を取得し、品質チェックを行い、OK/NG 判定を日本語で出力する。すべての出力は日本語で行うこと。

## 引数の処理

`$ARGUMENTS` を解析:

- `focus:<area>` → レビュー重点領域（例: `focus:security`, `focus:performance`）（任意）
- `spec:<path>` → 照合する Spec ファイルの絶対パス（任意）
- 引数なし → 汎用レビュー

## 処理フロー

1. Bash で `git diff` を取得（ステージ済み: `git diff --cached`, 未ステージ: `git diff`）
2. diff が空の場合は「レビュー対象の変更がありません。」と通知して終了
3. `spec:` が指定されている場合、Read で Spec ファイルを読み込み、`## 受入条件` を抽出
4. 以下の観点でレビューを実施

## レビュー観点

### 必須チェック（全レビューで実施）

| チェック項目 | 内容 |
|-------------|------|
| セキュリティ | ハードコードされたシークレット、SQL インジェクション、XSS、コマンドインジェクション |
| エラーハンドリング | 未処理の例外、エラーの握りつぶし |
| 型安全性 | any の乱用、型アサーションの不適切な使用（TypeScript の場合） |
| 命名規則 | 一貫性のない命名、略語の乱用 |
| デッドコード | 使用されていないインポート、到達不能コード |

### focus 指定時の追加チェック

| focus | 追加チェック |
|-------|-------------|
| security | 認証・認可のバイパス、入力バリデーション不足、CORS設定 |
| performance | N+1クエリ、不要な再レンダリング、メモリリーク |
| accessibility | aria 属性、キーボード操作、コントラスト |
| testing | テストカバレッジ、エッジケース、モック適切性 |

### Spec 照合（spec: 指定時）

Spec の受入条件の各項目について:
- 条件を満たす変更が diff に含まれているか → 合格
- 条件に関連する変更が不足している → 警告
- 条件に反する変更がある → 不合格

## 出力フォーマット

```
== コードレビュー結果 ==

判定: OK / NG

### サマリー
変更ファイル: N個
追加行: +N / 削除行: -N

### チェック結果
- [合格] セキュリティ: ハードコードされたシークレットなし
- [警告] エラーハンドリング: catch ブロックでエラーを無視している箇所あり (file.ts:42)
- [不合格] 型安全性: any が3箇所で使用 (file.ts:10, 25, 78)

### 受入条件の照合（Spec 指定時のみ）
- [合格] 条件1: ログインフォームが表示される
- [警告] 条件2: バリデーションメッセージの実装が不完全
- [不合格] 条件3: パスワードリセットリンクが未実装

### 推奨アクション
1. file.ts:42 のエラーハンドリングを修正
2. any を具体的な型に置換

判定根拠: 不合格が1件以上 → NG / 不合格なし → OK
```

## 判定ロジック

- **OK**: 不合格が0件（警告は許容）
- **NG**: 不合格が1件以上
