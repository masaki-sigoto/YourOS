# prompt-review スキル

ユーザーの過去のAIエージェント対話履歴を分析し、技術理解度・プロンプティングパターン・AI依存度を推定してレポートを生成する。レポートは日本語で `/Users/apple/YourOS/Scratch/Reviews/prompt-review-YYYY-MM-DD.md` に書き出す。

## 引数の処理

`$ARGUMENTS` を解析し、以下のルールで引数を処理する:

- 数値のみ → **日数フィルタ**（例: `30` → 過去30日分）
- 文字列のみ → **プロジェクト名フィルタ**（部分一致）
- 文字列 + 数値 → **プロジェクト名** + **日数フィルタ**（例: `yonshogen 30`）
- `tool:<name>` → **ツールフィルタ**（例: `tool:copilot`、`tool:claude`、`tool:cline`）
- 引数なし → 全プロジェクト横断、過去7日分（デフォルト）

例:
- `/prompt-review 30` → 過去30日分
- `/prompt-review homepage-renewal 14` → homepage-renewal プロジェクト、14日分
- `/prompt-review tool:copilot 30` → Copilot のみ、30日分

## ステップ1: データ収集（スクリプト実行）

前処理スクリプト [scripts/collect.py](scripts/collect.py) を実行してデータを収集する。
このスクリプトは Claude Code, GitHub Copilot Chat, Cline, Roo Code, Windsurf, Antigravity の
ログを自動検出し、フィルタ済みのJSON を標準出力に返す。

### 引数からスクリプトオプションを組み立てる

`$ARGUMENTS` を解析し、Bash で以下のように実行する:

```bash
python /Users/apple/.claude/skills/prompt-review/scripts/collect.py [OPTIONS] > /tmp/prompt-review-data.json
```

- 引数なし → オプションなし（デフォルト: 過去7日分）
- 数値のみ（例: `30`） → `--days 30`
- `all` または `0` → `--days 0`（全期間）
- 文字列のみ（例: `yonshogen`） → `--project yonshogen`
- 文字列 + 数値（例: `yonshogen 30`） → `--project yonshogen --days 30`

**重要**: スクリプトのパスは、必ず絶対パス `/Users/apple/.claude/skills/prompt-review/scripts/collect.py` を使うこと。現在の作業ディレクトリ（`cwd`）に関係なく、常にこの絶対パスを使用する。

### 出力の読み取り

スクリプト実行後、`/tmp/prompt-review-data.json` を Read で読み込む。

出力JSON構造:

```json
{
  "summary": {
    "total_messages": 2616,
    "detected_tools": ["Claude Code", "GitHub Copilot Chat"],
    "filter_days": null,
    "filter_project": null
  },
  "sources": [
    {
      "tool": "Claude Code",
      "status": "検出",
      "messages": [
        {"text": "プロンプト本文", "timestamp": "2025-09-29 03:16", "project": "yonshogen"}
      ],
      "period": "2025-09-29 03:16 〜 2026-03-12 04:58"
    }
  ],
  "project_stats": {
    "farbrain": {"count": 668, "tools": ["Claude Code"]},
    "yonshogen": {"count": 215, "tools": ["Claude Code"]}
  },
  "secret_warnings": []
}
```

### ツールフィルタの適用

`tool:` が指定されている場合、`sources` 配列から該当ツールのみをフィルタして分析対象とする:

| フィルタ値 | マッチするツール |
| ---------- | ---------------- |
| claude | Claude Code |
| copilot | GitHub Copilot Chat |
| cline | Cline |
| roo | Roo Code |
| windsurf | Windsurf |
| antigravity | Antigravity |

## ステップ2: 前回レポートの読み込み

Glob で `/Users/apple/YourOS/Scratch/Reviews/prompt-review-*.md` を検索し、最新のレポートファイルを取得。存在する場合は Read で読み込み、以下の情報を抽出:

- 前回のプロンプト品質スコア
- 前回の技術理解度マップ
- 前回の改善提案

この情報をステップ3の分析で「前回からの変化」として利用する。

## ステップ3: 分析

Read で `/tmp/prompt-review-data.json` を読み込んだら、以下の観点で `messages` 配列内のユーザープロンプトを分析する。各観点について**具体的なエビデンス**（実際のプロンプト断片の引用）を必ず含めること。

### 前処理: プロジェクト別サマリーの作成と短文応答の除外

まず `project_stats` を使ってプロジェクト別のメッセージ数・使用ツールの一覧を把握する。
各プロジェクトのプロンプト内容を読み、そのプロジェクトで行われている作業内容を1行で要約する。

次に、短文の肯定応答を分析対象から除外する。

### 前処理: 短文の肯定応答を無視する

Claude Code はユーザーに「〜しますか？」と確認を求めることが多く、それに対するユーザーの短い肯定応答はプロンプティング能力や技術理解度の分析に価値がない。以下のようなメッセージは分析対象から**除外**すること:

- 単純な肯定: 「y」「yes」「はい」「うん」「ok」「sure」「yep」「yeah」
- 実行指示: 「進めて」「やって」「do it」「doit」「go」「go ahead」「proceed」
- 承認: 「それで」「それでいい」「それでお願いします」「お願いします」「いいよ」「いいです」「大丈夫」
- 感謝のみ: 「ありがとう」「ありがとうございます」「thanks」「thx」

**判定基準**: メッセージが短文（概ね20文字以下）で、上記のパターンに該当するもの。ただし、短文でも具体的な技術指示を含むもの（例:「30pxがいいです」「asyncで」）は除外しない。

### 3a. プロンプト品質スコア（0-100）

以下の観点でスコアを算出:

| 観点 | 配点 | 評価基準 |
| ---- | ---- | -------- |
| 具体性 | 25点 | 抽象的な指示 vs 具体的な制約・条件の明示 |
| コンテキスト提供 | 25点 | 必要な背景情報や参照先を含んでいるか |
| 段階的指示 | 20点 | 複雑なタスクを分解して指示できているか |
| 出力形式の明示 | 15点 | 期待する結果の形式を伝えているか |
| 主体性 | 15点 | 方針決定を自分で行い、AIに実装を委任しているか |

スコアの目安:
- 80-100: エキスパート（AIを効果的に活用）
- 60-79: 上級（概ね適切だが改善余地あり）
- 40-59: 中級（基本はできるが具体性不足）
- 0-39: 初級（AIへの丸投げが多い）

### 3b. 技術的理解度マップ

プロンプトから言及されている技術概念を抽出し、3段階に分類:

- **熟知**: 自信を持って指示、正確な用語使用、具体的な実装方針の提示
- **基本理解**: 概念は知っているが詳細はAIに委任
- **学習中/曖昧**: 質問形式、誤解がある、試行錯誤が多い

分類のシグナル:
- 命令形で具体的な指示 → 熟知
- 「〜してください」＋概念名のみ → 基本理解
- 「〜って何？」「〜がうまくいかない」「どうすればいい？」 → 学習中

### 3c. プロンプティングパターン分析

- **効果的なパターン**: 具体的な制約指定、段階的な指示、十分なコンテキスト提供、期待する出力形式の明示
- **改善可能なパターン**: 曖昧な指示（「いい感じにして」等）、コンテキスト不足、目的と手段の混同
- **特徴的な癖**: 短い承認（「y」「doit」等）の頻度、日本語/英語の使い分け

### 3d. AI依存度分析

プロジェクト・ツールごとに:
- ユーザーが方針を決定しAIに実装を任せているケース（主体的）
- AIに方針決定から任せているケース（依存的）
- デバッグ/エラー解決でAIに頼るパターン
- 状態確認を頻繁にAIに依頼するパターン（「状況確認して」等）

### 3e. 成長の軌跡（前回比較を含む）

時系列で:
- プロンプトの質・具体性の変化
- 新しく使い始めた技術概念
- 繰り返し発生する課題パターン
- **前回レポートからの変化**: スコアの推移、新たに「熟知」に移行した技術、改善されたパターン

### 3f. プロジェクト横断・ツール横断の傾向

- 得意領域と不得意領域
- プロジェクト種類による振る舞いの違い
- AIツールの使い分け方（検出されたツールが複数の場合）

### 3g. シークレット・クレデンシャル警告

`secret_warnings` 配列が空でない場合、レポートの冒頭（データソースサマリーの直後）に警告セクションを出力する。

**レポート内ではシークレットの値は絶対に平文で書かない**。スクリプトがマスク済みの値（`masked_value`）を返すので、それをそのまま使う。

## ステップ4: アクショナブルな提案の生成

分析結果から、**次の1週間で実践できる具体的な改善アクション3つ**を生成する:

例:
1. 「コンテキスト提供を改善: プロンプトの冒頭に「目的: 〜」を1行追加する習慣をつけましょう。今週のプロンプトの40%でコンテキストが不足していました。」
2. 「段階的指示を実践: 複雑なタスクは「まず〜を調べて、次に〜を実装して」と分解して指示しましょう。」
3. 「エラー対応の自立: デバッグ時に「エラーメッセージ + 試したこと + 仮説」をセットで伝えると、AIの回答精度が上がります。」

## ステップ5: レポート生成

[references/report-template.md](references/report-template.md) のテンプレートに従い、日本語でレポートを生成する。以下のセクションを追加:

1. **プロンプト品質スコア** — レポート冒頭に大きく表示
2. **前回からの変化** — スコア推移、改善・悪化ポイント
3. **今週のアクション** — 3つの具体的改善提案

### 出力ルール

1. `/Users/apple/YourOS/Scratch/Reviews/` ディレクトリが存在しない場合は Bash で `mkdir -p /Users/apple/YourOS/Scratch/Reviews` を実行
2. Write ツールを使って `/Users/apple/YourOS/Scratch/Reviews/prompt-review-YYYY-MM-DD.md` に書き出す（YYYYMMDDは実行日）
3. レポート生成後、ファイルパスとスコアをユーザーに通知する

### 記述上の注意

- レポートは**必ず日本語**で出力すること
- 推測は「〜と推測されます」「〜の可能性があります」と明示すること
- プロンプトの原文を引用する場合は**短く切り取り**、プライバシーに配慮すること
  - ファイルパスの個人名部分は `<user>` にマスク
  - プロジェクト固有の機密情報は伏せる
- 分析できなかった部分は「データ不足のため分析できませんでした」と正直に記載
- どのツールからの情報かを括弧書きで明記する（例:「(Claude Code)」「(Copilot Chat)」）
- エビデンスのないセクションは省略してよい
